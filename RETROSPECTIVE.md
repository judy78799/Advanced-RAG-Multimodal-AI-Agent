# Naive RAG 구현 회고록

## 1. 시작 배경

이번 실습의 목표는 네이버 2026년도 1분기 실적 발표 PDF를 이용해 가장 기본적인 RAG 파이프라인을 직접 구현해보는 것이었다.

처음에는 다음 흐름을 하나씩 확인하는 데 집중했다.

- Document Loader로 PDF 문서 불러오기
- Text Splitter로 문서 청크 분할하기
- Embedding 적용하기
- Vector DB 생성하기
- Retriever로 관련 문서 검색하기
- 검색된 문서를 기반으로 LLM 답변 생성하기

구현 대상 문서는 `docs/1Q26_Naver_Earnings_KOR_vF.pdf`였다. 재무 실적 발표 자료이기 때문에 표, 그래프, 숫자, 단위가 섞여 있어 RAG의 장점과 한계를 함께 확인하기 좋은 자료였다.

## 2. 구현 과정

먼저 `src/naive_rag.py` 파일에 기본 RAG 파이프라인을 구성했다.

PDF 로딩에는 LangChain의 `PyPDFLoader`를 사용했다. 로더를 통해 PDF 페이지별 텍스트를 추출했고, 실제로 12페이지가 정상적으로 로드되는 것을 확인했다.

이후 `RecursiveCharacterTextSplitter`를 사용해 문서를 청크로 나누었다. 청크 크기는 `900`, overlap은 `150`으로 설정했다. 네이버 실적 발표 PDF는 페이지 수가 많지 않고 표나 그래프 주변 텍스트가 비교적 짧기 때문에, 처음 실험용으로는 과하게 작은 청크보다 문맥을 어느 정도 유지하는 설정이 적절하다고 판단했다.

Embedding은 OpenAI의 `text-embedding-3-small`을 사용했다. Vector DB는 Chroma를 사용했고, 저장 경로는 `data/chroma_naver_1q26`으로 지정했다. 이렇게 한 번 생성한 DB는 이후 질문에서 재사용할 수 있도록 했다.

마지막으로 Retriever가 가져온 문서를 `ChatOpenAI` 모델에 넘겨 답변을 생성하도록 구성했다. 프롬프트에는 다음 조건을 넣었다.

- 주어진 context에 있는 내용만 사용할 것
- 숫자, 기간, 사업부 이름을 정확히 적을 것
- 근거 페이지를 표시할 것
- context에 없는 내용은 모른다고 답할 것

## 3. 청킹 전략

이번 실습에서는 `RecursiveCharacterTextSplitter`를 사용해 문서를 청크로 나누었다.

설정은 다음과 같았다.

```python
RecursiveCharacterTextSplitter(
    chunk_size=900, # 700/500
    chunk_overlap=150, # 50/100
    separators=["\n\n", "\n", ". ", " ", ""],
)
```

이 설정을 선택한 이유는 실적 발표 PDF의 특성 때문이다.

네이버 실적 발표 자료는 긴 문단 중심 문서라기보다, 한 페이지 안에 제목, 표, 그래프 숫자, 설명 bullet이 함께 들어 있는 형태다. 따라서 청크를 너무 작게 나누면 표의 헤더와 숫자, 그래프 제목과 값, 설명 문장이 서로 분리될 가능성이 있다.

예를 들어 파이낸셜 플랫폼 페이지에는 다음 정보가 함께 등장한다.

- `분기 별 매출`
- `4Q24 ~ 1Q26`
- `395.1, 386.7, 405.6, 427.3, 448.6, 459.7`
- `단위: 십억 원, 조 원`
- `1Q26 YoY 18.9%, QoQ 2.5%`
- 결제액, 외부 결제액 관련 설명

이 정보들이 서로 다른 청크로 갈라지면 Retriever가 일부 숫자만 가져오거나, 단위 없이 값만 가져올 수 있다. 그래서 처음에는 한 페이지의 주요 문맥이 어느 정도 유지되도록 `chunk_size=900`을 사용했다.

`chunk_overlap=150`은 청크 경계에서 중요한 문장이 잘리는 문제를 줄이기 위한 설정이다. 특히 PDF에서 줄바꿈이 많고 표 형태의 텍스트가 연속적으로 추출되기 때문에, overlap이 없으면 분기명과 숫자 값이 서로 다른 청크로 분리될 수 있다.

현재 전략의 장점은 다음과 같다.

- 구현이 단순하다.
- 작은 PDF에서 빠르게 실험하기 좋다.
- 그래프 제목, 숫자, 설명 문장이 같은 청크에 남을 가능성이 높다.
- Retriever가 관련 페이지를 찾기 쉽다.

하지만 한계도 있다.

- 표 구조를 이해해서 나누는 것이 아니라 단순 문자 기준으로 나눈다.
- 그래프 하나, 표 하나를 의미 단위로 정확히 보존하지 못한다.
- 비슷한 숫자가 여러 페이지에 있으면 검색 결과가 섞일 수 있다.
- 단위와 숫자가 같은 청크에 있더라도 LLM이 단위를 잘못 해석할 수 있다.
- PDF 텍스트 추출 순서가 어긋나면 청킹 품질도 함께 떨어진다.

다음 단계에서는 청킹 전략을 다음과 같이 개선해볼 수 있다.

- 페이지 단위 청킹: 실적 발표 자료처럼 한 페이지가 하나의 슬라이드인 PDF에 적합하다.
- Slide-aware chunking: PDF의 각 페이지를 하나의 슬라이드로 보고, 슬라이드 제목, 본문 bullet, 표, 그래프 영역을 구분해 청크를 구성한다.
- 섹션 단위 청킹: 제목, 표, bullet 설명을 기준으로 의미 단위를 나눈다.
- 표 전용 청킹: 표는 행과 열 구조를 유지한 별도 문서로 저장한다.
- 그래프 전용 청킹: 그래프 제목, 축 라벨, 범례, 숫자 라벨을 묶어 하나의 청크로 저장한다.
- 메타데이터 강화: `page`, `section_title`, `chart_title`, `unit` 같은 정보를 metadata에 넣는다.
- Parent-child chunking: 작은 청크로 검색하되, 답변에는 해당 페이지 전체나 상위 섹션을 함께 제공한다.

이번 실습에서는 단순한 문자 기반 청킹만으로도 꽤 좋은 결과가 나왔지만, 이는 PDF의 텍스트 추출 상태가 좋았기 때문이다. 다음 단계에서는 특히 `slide-aware chunking`을 적용해볼 예정이다. 네이버 실적 발표 자료처럼 한 페이지가 하나의 슬라이드로 구성된 PDF에서는 단순 문자 길이 기준보다 슬라이드 구조를 인식하는 방식이 더 적합하다.

`slide-aware chunking`에서는 각 슬라이드를 하나의 상위 문맥으로 유지하면서, 내부적으로는 제목, 핵심 bullet, 표, 그래프 영역을 분리해 metadata를 붙일 수 있다. 이렇게 하면 Retriever가 단순히 비슷한 문장을 찾는 것을 넘어, "파이낸셜 플랫폼 그래프", "실적요약 표", "비용/손익 슬라이드"처럼 문서 구조에 가까운 단위로 검색할 수 있다.

재무 문서 분석의 신뢰도를 높이려면 앞으로는 숫자와 단위, 표와 그래프의 구조를 보존하는 방향으로 청킹 전략을 개선해야 한다.

## 4. 노트북으로 재구성

스크립트로 전체 흐름을 먼저 구현한 뒤, 같은 내용을 `notebooks/rag.ipynb`로도 정리했다.

노트북은 학습 흐름을 보기 쉽게 다음 단계로 나누었다.

2. PDF 로드
3. 청크 분할
4. PDF 텍스트 추출 상태 확인
5. Embedding 생성
6. Chroma Vector DB 생성
7. Retriever 검색 결과 확인
8. LLM 답변 생성

스크립트는 실행용이고, 노트북은 학습과 디버깅용이라는 역할을 나누었다. 특히 노트북에서는 Retriever가 실제로 어떤 페이지와 청크를 가져오는지 직접 확인할 수 있게 해두었다.

## 5. 그래프 질의 테스트

Naive RAG가 PDF 안의 막대그래프 관련 질문에 얼마나 잘 답하는지 확인하기 위해 다음과 같은 질문을 던졌다.

```text
파이낸셜 플랫폼 막대 그래프에서 분기 별 매출을 분석해줘
```

결과는 예상보다 좋았다. 답변은 파이낸셜 플랫폼의 분기별 매출을 다음과 같이 정리했다.

| 분기 | 매출 |
|---|---:|
| 4Q24 | 395.1 |
| 1Q25 | 386.7 |
| 2Q25 | 405.6 |
| 3Q25 | 427.3 |
| 4Q25 | 448.6 |
| 1Q26 | 459.7 |

처음에는 Naive RAG가 막대그래프를 꽤 잘 이해하는 것처럼 보였다. 하지만 확인해보니 실제 이유는 다소 달랐다.

PDF의 막대그래프 안 숫자와 축 라벨이 이미지로만 존재한 것이 아니라, 텍스트로도 추출되고 있었다. 즉, 모델이 그래프 이미지를 시각적으로 읽은 것이 아니라 `PyPDFLoader`가 추출한 텍스트를 검색하고 정리한 것이다.

이 지점이 이번 실습에서 가장 중요한 깨달음이었다.

## 6. 발견한 한계

이번 테스트에서 Naive RAG의 장점뿐 아니라 한계도 확인할 수 있었다.

첫 번째 한계는 그래프 이미지를 직접 이해하지 못한다는 점이다.

현재 구조는 멀티모달 RAG가 아니다. PDF에서 추출된 텍스트만 Vector DB에 저장된다. 따라서 막대 높이, 색상, 범례, 선 그래프와 막대그래프의 관계처럼 시각 정보에만 존재하는 내용은 안정적으로 분석할 수 없다.

두 번째 한계는 단위 해석 오류다.

테스트 답변에서 실제로 단위 오류가 발생했다. 원문은 `단위: 십억 원`이었지만, 답변에서는 일부 값을 `억 원`으로 표현했다. 예를 들어 `459.7`은 원문 기준으로 `459.7십억 원`, 즉 `4,597억 원`에 해당한다. 하지만 LLM은 이를 `459.7억 원`처럼 잘못 표현할 수 있다.

세 번째 한계는 검색이 맞아도 생성 답변이 틀릴 수 있다는 점이다.

Retriever가 관련 페이지를 잘 가져오더라도, 최종 답변은 LLM이 생성한다. 이 과정에서 단위 변환, 계산, 요약, 비교가 들어가면 오류 가능성이 생긴다. RAG는 검색 정확도와 생성 정확도를 모두 확인해야 한다.

네 번째 한계는 표와 그래프의 범위 차이를 혼동할 수 있다는 점이다.

예를 들어 실적요약 표는 `1Q25~1Q26` 범위를 담고 있고, 파이낸셜 플랫폼 그래프는 `4Q24~1Q26` 범위를 담고 있다. 질문이 모호하면 Retriever가 표와 그래프 정보를 함께 가져오고, LLM이 기간 범위를 혼동할 수 있다.

다섯 번째 한계는 계산형 질문에 취약하다는 점이다.

값을 단순히 나열하는 것은 비교적 잘하지만, 증가액, 증가율, 최대 증가 구간을 계산하라고 하면 오류 가능성이 커진다. Naive RAG에는 계산 검증 로직이 없기 때문이다.

## 7. 좋았던 점

이번 실습에서 좋았던 점은 RAG의 전체 흐름을 빠르게 관통해볼 수 있었다는 것이다.

PDF 하나를 대상으로 로더, splitter, embedding, vector DB, retriever, LLM 답변 생성까지 연결해보니 RAG가 실제로 어떻게 작동하는지 감이 잡혔다.

또한 Retriever가 어떤 청크를 가져오는지 직접 확인하면서, RAG 성능은 단순히 LLM 모델 성능만의 문제가 아니라는 점을 체감했다. 문서 로딩 품질, 청크 크기, 질문 표현, 검색된 문맥의 품질이 모두 최종 답변에 영향을 준다.

특히 그래프 질의 테스트를 통해 "잘 맞는 답변"과 "진짜로 이해한 답변"은 다를 수 있다는 점을 확인했다. 겉보기 성능이 좋아도, 내부적으로는 텍스트 추출에 의존하고 있을 수 있다.

## 8. 아쉬웠던 점

아쉬웠던 점은 아직 이미지 기반 이해가 들어가지 않았다는 것이다.

이번 구조에서는 PDF 페이지를 이미지로 렌더링하지 않았고, Vision 모델도 사용하지 않았다. 그래서 그래프가 텍스트로 추출되지 않는 PDF라면 같은 질문에 제대로 답하지 못할 가능성이 높다.

또한 단위 검증이나 계산 검증을 자동화하지 않았다. 재무 문서에서는 단위 하나가 틀려도 답변의 신뢰도가 크게 떨어진다. 다음 단계에서는 단위와 숫자를 더 엄격하게 다루는 구조가 필요하다.

## 9. 다음 개선 방향

다음 단계에서는 Naive RAG를 다음 방향으로 개선하고 싶다.

- 검색된 원문 청크를 답변과 함께 출력하기
- 답변 전 단위 검증 프롬프트 강화하기
- 계산은 LLM이 직접 하지 않고 Python 함수로 처리하기
- Slide-aware chunking을 적용해 슬라이드 제목, 표, 그래프, bullet을 구조화하기
- PDF 표 구조를 별도로 추출하기
- PDF 페이지를 이미지로 렌더링해 Vision 모델에 입력하기
- 텍스트 RAG와 이미지 RAG를 결합한 Multimodal RAG로 확장하기
- 질문별로 Retriever 결과가 적절했는지 평가하는 테스트셋 만들기

특히 재무 PDF 분석에서는 표와 그래프가 핵심이므로, 단순 텍스트 RAG만으로는 한계가 있다. 다음 목표는 텍스트로 추출된 정보와 이미지에서 읽은 정보를 함께 사용하는 구조를 만드는 것이다.

## 10. 정리

이번 실습을 통해 Naive RAG는 빠르게 구현할 수 있고, 텍스트 추출이 잘 되는 PDF에서는 꽤 좋은 결과를 낼 수 있다는 것을 확인했다.

하지만 동시에 다음 사실도 분명해졌다.

- Naive RAG는 그래프 이미지를 직접 이해하지 않는다.
- PDF 텍스트 추출 품질에 크게 의존한다.
- 검색이 성공해도 생성 답변에서 단위나 계산 오류가 날 수 있다.
- 재무 문서에서는 단위, 기간, 표/그래프 범위를 별도로 검증해야 한다.

결론적으로 Naive RAG는 좋은 출발점이지만, 신뢰할 수 있는 재무 문서 분석 시스템으로 가기 위해서는 멀티모달 처리, 구조화된 표 추출, 계산 검증이 추가되어야 한다.

이번 실습은 그 한계를 직접 확인했다는 점에서 의미가 있었다.
